### MyBDR에 없는 것 (BDR-join-v1에만 있는 핵심 기능)

| 기능 | 설명 | 중요도 |
|------|------|--------|
| **대회 참가신청 폼** | 팀 단위 대회 등록 위자드 | 핵심 |
| **부문/디비전 체계** | 일반부-D3/D4, 대학부-U3 등 카테고리 분류 | 핵심 |
| **부문별 정원 관리** | 디비전당 최대 팀 수, 잔여 석 표시 | 핵심 |
| **대기접수** | 정원 초과 시 대기 → 자동 전환 | 핵심 |
| **유니폼 색상 등록** | 홈/어웨이 컬러 + 유니폼 프리뷰 | 중요 |
| **선수 명단 일괄 등록** | 붙여넣기로 5명 이상 한번에 등록 | 중요 |
| **참가비 안내/입금 확인** | 계좌 정보 표시, 입금 상태 관리 | 핵심 |
| **신청서 조회 (비로그인)** | 이름+연락처로 신청 현황 확인 | 낮음 |

## 2. 구현 범위 정의

> 원칙: MyBDR은 **로그인 기반 플랫폼**이므로 BDR-join-v1의 "비로그인 비밀번호 방식"은 제거한다.
> 원칙: 기존 MyBDR의 팀/프로필 데이터를 **재활용**하여 중복 입력을 최소화한다.

## 3. 대회 관리 파트 (개최자)

> 경로: `/tournament-admin/tournaments/[id]/wizard` 확장

### 3-1. 대회 생성 위자드에 추가할 단계

현재 위자드: 기본 정보 → 일정/장소 → 팀 설정 → 규칙/상금 → 디자인

**추가할 단계: "접수 설정" (Step 3으로 삽입, 기존 3~5는 4~6으로 밀림)**

#### 3-1-A. 부문/디비전 설정

```
필드: categories (JSONB)
구조: {
  "일반부": ["D3", "D4", "D5"],
  "대학부": ["U3"],
  "여성부": ["W4"]
}
```

| 항목 | 설명 |
|------|------|
| 부문(Category) 추가/삭제 | "일반부", "대학부", "여성부", "유소년부" 등 자유 입력 |
| 디비전(Division) 추가/삭제 | 각 부문 하위에 D3, D4, U3 등 배치 |
| 디비전별 정원 | `div_caps: { "D3": 16, "D4": 24 }` |
| 디비전별 참가비 | `div_fees: { "D3": 350000, "D4": 320000 }` (선택) |

> **왜 기존 `maxTeams`와 별개인가:**
> 현재 `maxTeams`는 대회 전체 정원이다. 부문/디비전 체계에서는 각 디비전이 독립된 토너먼트처럼 운영되므로, 디비전별 정원이 필요하다. `maxTeams`는 전체 합산 정원으로 자동 계산되도록 한다.

#### 3-1-B. 접수 기간 설정

| 필드 | 타입 | 설명 |
|------|------|------|
| `registration_start_at` | DateTime | 접수 시작일시 |
| `registration_end_at` | DateTime | 접수 마감일시 |
| `allow_waiting_list` | Boolean | 대기접수 허용 여부 |
| `waiting_list_cap` | Int | 대기 최대 수 (디비전당) |

> 이 필드들은 이미 Prisma 스키마에 존재하지만 (`registration_start_at`, `registration_end_at`) 위자드 UI에서 노출되지 않고 있다. UI만 추가하면 된다.

#### 3-1-C. 참가비/입금 정보

| 필드 | 타입 | 설명 |
|------|------|------|
| `entry_fee` | Decimal | 기본 참가비 (이미 존재) |
| `bank_name` | String | 입금 은행명 |
| `account_number` | String | 계좌번호 |
| `account_holder` | String | 예금주 |
| `fee_notes` | String | 참가비 관련 안내 (환불 정책 등) |

> 개최자의 계좌 정보는 대회별로 다를 수 있으므로 대회 테이블에 저장한다.
> 유저 프로필의 bank_info와는 다른 용도 (프로필은 정산용, 여기는 참가비 수납용).

### 3-2. 팀 관리 확장 (개최자 화면)

> 경로: `/tournament-admin/tournaments/[id]/teams` 확장

현재: 승인/거절 + 시드 배정만 가능

**추가 기능:**

| 기능 | 설명 |
|------|------|
| **입금 상태 관리** | 각 팀별 `payment_status` 토글: 미입금/입금완료/환불 |
| **대기 → 정규 전환** | 정원에 빈자리 발생 시 대기팀을 정규로 올리는 버튼 |
| **선수 명단 열람** | 등록된 선수 명단 확인 (등번호, 포지션, 생년월일, 선출 여부) |
| **디비전별 필터** | 팀 목록을 부문/디비전으로 필터링 |
| **참가 현황 요약** | 디비전별 등록/정원/대기 수 카드 |

## 4. 유저 파트 (참가자)

### 4-1. 대회 참가신청 위자드

> 경로: `/tournaments/[id]/join`
> 접근: 로그인 필수

**BDR-join-v1과의 차이점:**
- BDR-join-v1: 비로그인, 비밀번호 방식, 팀 정보 처음부터 입력
- MyBDR: 로그인 기반, **기존 팀 선택** 또는 새 팀 생성, 프로필 데이터 자동 연동

#### Step 1: 팀 선택

| 항목 | 설명 |
|------|------|
| 내 팀 목록 | 유저가 주장인 팀 목록에서 선택 |
| 새 팀으로 참가 | 즉석에서 대회 전용 팀 생성 (팀명 한/영만 입력) |
| 대표자 정보 | 프로필에서 자동 로드 (이름, 연락처) — 수정 가능 |

> **왜 기존 팀을 선택하게 하는가:**
> MyBDR에는 이미 팀 시스템이 있다. BDR-join-v1처럼 매번 팀 정보를 새로 입력하면 데이터가 파편화된다.
> 기존 팀을 선택하면 멤버 명단도 자동으로 불러올 수 있어 입력을 최소화할 수 있다.

#### Step 2: 부문/디비전 선택

| 항목 | 설명 |
|------|------|
| 부문 선택 | 대회에 설정된 카테고리 중 택 1 |
| 디비전 선택 | 해당 부문의 디비전 중 택 1 |
| 잔여 정원 | 실시간 표시 (예: "D4 — 14/16팀") |
| 대기접수 표시 | 정원 초과 시 "대기접수" 배지 + 대기 순번 안내 |
| 참가비 | 디비전별 참가비 표시 |

#### Step 3: 유니폼 / 선수 등록

| 항목 | 설명 |
|------|------|
| 유니폼 색상 | 홈/어웨이 컬러 픽커 (SVG 프리뷰는 향후 옵션) |
| 선수 명단 | 팀 선택 시 기존 멤버 자동 로드 → 체크박스로 참가 선수 선택 |
| 선수 추가 | 팀 멤버가 아닌 선수 추가 가능 (이름, 등번호, 포지션, 생년월일, 선출) |
| 최소 인원 검증 | 대회 설정의 `roster_min` 이상 등록 필수 |
| 등번호 중복 검증 | 같은 팀 내 등번호 중복 방지 |

> **핵심 차별점:**
> BDR-join-v1은 선수를 매번 수동 입력한다. MyBDR은 팀 멤버를 불러와서 **체크만 하면 되므로** 훨씬 빠르다.
> 팀 멤버가 아닌 용병은 별도 추가 가능하게 하되, 이 경우 `auto_registered: false`로 표시.

#### Step 4: 확인 및 제출

| 항목 | 설명 |
|------|------|
| 신청 요약 | 팀명, 부문/디비전, 선수 수, 참가비 |
| 입금 안내 | 계좌 정보 + 복사 버튼 |
| 약관 동의 | 환불 정책 등 |
| 제출 | `tournament_teams` + `tournament_team_players` 생성 |
| 완료 화면 | 신청 번호, 입금 안내, "내 신청 보기" 링크 |

### 4-2. 내 대회 신청 현황

> 경로: `/profile` 내 기존 "대회" 섹션 확장 또는 `/tournaments/my`

| 항목 | 설명 |
|------|------|
| 신청 목록 | 내가 신청한 모든 대회 (상태별 필터: 대기/승인/반려) |
| 상태 확인 | 접수상태 + 입금상태 배지 |
| 신청서 수정 | 승인 전이면 선수 명단/유니폼 수정 가능 |
| 신청 취소 | 승인 전이면 취소 가능 |

> **BDR-join-v1의 `/lookup` 페이지는 불필요:**
> MyBDR은 로그인 기반이므로 비로그인 조회 기능이 필요 없다.
> 프로필 페이지의 대회 섹션에서 동일한 기능을 제공한다.

### 4-3. 대회 상세 페이지 확장

> 경로: `/tournaments/[id]` (기존 페이지 확장)

현재: 대회 정보 + 참가팀 + 대진표

**추가할 내용:**

| 항목 | 설명 |
|------|------|
| 접수 상태 배지 | "접수중" / "마감임박" / "마감" / "대기접수" |
| 부문/디비전 정원 | 디비전별 등록현황 카드 |
| 참가비 정보 | 금액 + 계좌 정보 |
| 참가신청 버튼 | 접수 기간 내 + 로그인 상태에서만 활성 |
| 등록된 팀 목록 | 디비전별 그룹핑, 팀명 표시 |
| 내 신청 상태 | 이미 신청한 경우 현재 상태 표시 |

### 4-4. 프로필 연동

**이미 충분한 부분 (추가 불필요):**
- 포지션, 키, 지역, 연락처 → 대회 신청 시 자동 로드
- 팀 목록 → 신청 시 팀 선택에 활용
- 스탯 (득점, 리바운드, 어시스트) → 대회 결과에서 자동 집계 (이미 구현)

**자연스럽게 연동되는 흐름:**
```
프로필 완성 → 팀 가입/생성 → 대회 참가 신청 (팀+멤버 자동 로드)
                                  ↓
                           대회 진행 → 스탯 기록
                                  ↓
                           프로필 스탯 바에 반영 (이미 구현)
```

## 5. 구현하지 않을 것 (중복/불필요)

| BDR-join-v1 기능 | 제외 이유 |
|------------------|-----------|
| 비밀번호 기반 신청서 관리 | MyBDR은 로그인 기반이므로 불필요 |
| `/lookup` 신청서 조회 | 프로필 내 대회 섹션으로 대체 |
| 팀 정보 매번 수동 입력 | 기존 팀 시스템 활용으로 자동화 |
| 선수 정보 매번 수동 입력 | 팀 멤버 데이터에서 불러와서 체크 |
| Google Sheets 연동 | Prisma DB 직접 관리로 대체 |
| Solapi SMS 알림 | 앱 내 알림 시스템(notifications 테이블)으로 대체 |
| TMap 장소 검색 | 현재 텍스트 입력 방식 유지 (지도는 별도 로드맵) |
| `legacy/` HTML 파일 | 완전히 폐기 |
| 대회 프리셋 (tournament_presets) | 시리즈/에디션 기능으로 대체 (이미 구현) |
| 관리자 대시보드 차트 | MyBDR 어드민 대시보드에 이미 존재 |
| 지역(시/도, 시/군/구) 선택 | MyBDR 프로필/팀에 이미 구현 |
| 유니폼 SVG 프리뷰 | 초기에는 컬러 칩만 표시, 프리뷰는 Phase 2 |

## 6. 구현 우선순위

### Phase 1: 핵심 흐름 (대회 참가의 처음부터 끝까지)

```
[1] DB 스키마 확장 (tournaments, tournament_teams, tournament_team_players)
[2] 대회 위자드 "접수 설정" 단계 추가 (부문/디비전, 접수기간, 참가비)
[3] 대회 참가신청 위자드 (/tournaments/[id]/join) — 4단계
[4] 대회 상세 페이지 확장 (접수 상태, 부문 정원, 참가 버튼)
[5] 팀 관리 확장 (입금 상태, 디비전 필터, 선수 명단 열람)
[6] 프로필 내 대회 신청 현황
```

### Phase 2: 운영 편의 (대회 운영 효율화)

```
[7] 대기접수 → 정규 전환 자동화 + 알림
[8] 유니폼 색상 SVG 프리뷰
```

## 9. 결론

BDR-join-v1의 핵심 가치는 **"대회 참가신청 위자드"**와 **"부문/디비전 체계"** 두 가지다.

나머지 기능(팀 관리, 선수 등록, 프로필, 관리자 대시보드)은 MyBDR에 이미 더 나은 형태로 존재한다.

통합 시 BDR-join-v1의 코드를 직접 마이그레이션하지 않고, 개념과 UX 흐름만 참고하여 MyBDR의 기존 아키텍처(Prisma + NextAuth + Zod) 위에 새로 구현하는 것이 올바른 접근이다.

**최종 신규 작업량 추정:**
- DB 스키마: 3개 테이블 확장
- API 엔드포인트: 4~5개 신규
- 페이지: 2개 신규 (참가신청 위자드, 대회 상세 확장)
- 기존 페이지 수정: 3개 (대회 위자드, 팀 관리, 프로필 대회 섹션)
